**Nested If**

## 1. Introduction

Nested if statements place one `if-else` block inside another, allowing multi-level decision making. They exist to handle hierarchical conditions common in real data and business logic. While powerful, they can quickly reduce readability. In ML engineering, nested ifs appear in feature engineering, validation rules, and conditional pipeline routing, but experienced engineers prefer flatter alternatives when possible.

## 2. Core Theory

- **Definition**: An `if` statement (or `elif`/`else`) contained within another `if` block.
- **Syntax**:
  ```python
  if outer_condition:
      if inner_condition:
          # code
      else:
          # code
  ```
- **Key Properties**:
  - Increases cyclomatic complexity.
  - Each level adds indentation and cognitive load.
  - Short-circuiting still applies at every level.
  - Prone to bugs if not carefully managed.

## 3. Core Methods / Operations

**Basic Nested If**
```python
def process_sample(features: dict, threshold: float = 0.8) -> str:
    if "age" in features:
        if features["age"] > 65:
            if features.get("income", 0) > 100000:
                return "High Value Senior"
            else:
                return "Senior"
```

**Flattened with Guard Clauses** (preferred)
```python
def process_sample(features: dict, threshold: float = 0.8) -> str:
    if "age" not in features:
        return "Invalid"
    if features["age"] <= 65:
        return "Standard"
    if features.get("income", 0) > 100000:
        return "High Value Senior"
    return "Senior"
```

**Combined with Logical Operators**
```python
if "age" in features and features["age"] > 65 and features.get("income", 0) > 100000:
    ...
```

## 4. Production Use Cases

1. **Backend APIs**: Multi-level input validation and authorization.
2. **Automation Scripts**: Environment + mode + resource checks.
3. **Data Processing**: Conditional feature derivation based on multiple attributes.
4. **Machine Learning**: Staged preprocessing (data quality → cleaning → augmentation).
5. **Production Software**: Fallback logic in inference serving (model A → model B → default).

## 5. Machine Learning Connection

- **Pandas**: Nested conditions in `apply()` or custom transformers.
- **Scikit-learn / Pipelines**: Conditional steps based on data characteristics.
- **PyTorch**: Dynamic computation graphs, mixed precision, or device-specific logic.
- **Experiment Management**: Different training strategies based on dataset size + model type + hardware.

## 6. Common Mistakes

1. **Excessive nesting depth**  
   Wrong: 4+ levels deep.  
   Correct: Refactor to guard clauses or helper functions.  
   Why: Poor readability and maintainability.

2. **Repeated conditions**  
   Wrong: Checking same key at multiple levels.  
   Correct: Extract to variables first.

3. **Missing else clauses at inner levels**  
   Wrong: Silent fall-through.  
   Correct: Explicit handling or logging.

4. **Deep nesting in performance-critical loops**  
   Wrong: Nested ifs inside training data loader.  
   Correct: Pre-compute or vectorize with NumPy/Pandas.

5. **Hard-to-test complex nested logic**  
   Wrong: Large nested block.  
   Correct: Break into small, testable functions.

def route_inference(sample: dict, model_registry: dict) -> str:

**4. Real Software Example**
```python
def preprocess_feature(value: any, feature_type: str, stats: dict) -> float:
    if value is None:
        return stats.get("mean", 0.0)
    if feature_type == "numeric":
        if value < stats.get("min", 0):
            return stats["min"]
        elif value > stats.get("max", 1):
            return stats["max"]
    return float(value)
```

## 8. Time Complexity

| Operation             | Time Complexity |
|-----------------------|-----------------|
| Simple nested if      | O(1)            |
| Deep nesting          | O(d) (depth)    |
| Per evaluation        | Depends on conditions |

## 9. Key Takeaways

- Favor flat structures and guard clauses over deep nesting.
- Extract complex conditions into well-named helper functions.
- Use logical operators (`and`/`or`) to reduce nesting when appropriate.
- Limit nesting depth to 2-3 levels maximum in production code.
- Pre-compute common checks before nested blocks.
- Nested ifs are acceptable for clear hierarchical logic but should be refactored as code grows.
- In ML, prefer declarative approaches (config-driven) over deeply nested conditionals.
- Always ensure all paths are handled and testable.
- Master refactoring nested ifs — it separates junior from senior ML engineers.
