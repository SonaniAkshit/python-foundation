**Topic**

**Nested Conditional Statements (Nested if)**

---

## 1. Introduction

Nested `if` statements occur when you place one `if-elif-else` block inside another. They allow multi-level decision making where an outer condition must be true before inner conditions are evaluated.

Python supports nesting because complex real-world logic often requires hierarchical decisions (e.g., first check if user is authenticated, then check permissions, then check resource state). They solve problems involving **dependent conditions** that cannot be easily flattened into a single chain.

However, they are a double-edged sword: powerful but dangerous for readability if overused.

---

## 2. Intuition

Imagine a **job interview process** for your Resume Intelligence system:
- Outer check: Does the candidate have the minimum years of experience? (Gatekeeper)
  - If yes → proceed to inner checks:
    - Do they have Python skills?
      - If yes → check ML experience?
        - If yes → "Strong Match"
        - Else → "Potential with upskilling"

Without nesting, you would need many complex boolean expressions that become hard to read and debug.

---

## 3. Syntax

```python
if outer_condition:
    if inner_condition1:
        # deepest block
        ...
    elif inner_condition2:
        ...
    else:
        ...
elif outer_condition2:
    ...
else:
    ...
```

- Indentation level increases with each nesting.
- You can nest as deep as needed (but shouldn't).
- Combine with `and`/`or` when possible to reduce depth.

**Early return pattern** (preferred alternative):
```python
def evaluate(...):
    if not outer_condition:
        return "Failed gate"
    if not inner_condition:
        return "Failed secondary check"
    # happy path
    return "Success"
```

---

## 4. Internal Working

Each nested level adds another conditional jump in the bytecode. Python evaluates from outer to inner. If an outer condition is `False`, all inner blocks are skipped entirely (efficient short-circuit at the structural level).

Memory impact is negligible — it's purely control flow. Deep nesting can make stack traces harder to read during debugging.

---

## 5. Built-in Methods / Operations

No new built-ins, but key patterns:

- **Guard clauses** (flatten nesting)
- **Helper functions** to extract inner logic
- `match-case` (Python 3.10+) as modern alternative for some nested cases

**Practical example**:
```python
def assess_resume_fit(resume: dict, job: dict) -> dict:
    # Level 1: Basic eligibility
    if resume.get("years_exp", 0) < job.get("min_exp", 0):
        return {"status": "Rejected", "reason": "Insufficient experience"}
    
    # Level 2: Skill match
    if not has_required_skills(resume["skills"], job["required_skills"]):
        return {"status": "Rejected", "reason": "Missing core skills"}
    
    # Level 3: Quality deep dive
    score = calculate_score(resume, job)
    if score >= 85:
        recommendation = "Strong Hire"
    elif score >= 70:
        recommendation = "Interview"
    else:
        recommendation = "Consider"
    
    return {"status": "Qualified", "score": score, "recommendation": recommendation}
```

## 6. Common Mistakes

**Mistake 1**: Excessive nesting (pyramid of doom)
```python
if a:
    if b:
        if c:
            if d:   # Hard to follow
```

**Mistake 2**: Repeating the same outer checks in multiple places.

**Mistake 3**: Mixing business logic with validation in deep nests → untestable code.

**Correct approach**: Use early returns, extract methods, or restructure into a decision tree / strategy pattern.

---

## 7. Best Practices

- Limit nesting to 2-3 levels maximum.
- Extract inner logic into well-named functions.
- Prefer flat structures with guard clauses.
- Use comments sparingly — let clean code speak.
- Consider polymorphism or dictionaries of handlers for very complex branching.
- Always validate inputs at the earliest level.

## 10. Resume Intelligence Project Integration

**Feature to add**: Multi-stage Resume Screening Engine with hierarchical evaluation.

**Why it belongs**: Real hiring involves layered decisions — experience gate → skills validation → cultural/technical depth → final ranking. Nesting (or better, structured alternatives) naturally models this.

**Example implementation idea**:
```python
class ResumeScreener:
    def screen(self, resume: dict, job: dict) -> dict:
        # Stage 1: Hard filters
        if not self._meets_hard_requirements(resume, job):
            return self._rejection("Hard requirements failed")
        
        # Stage 2: Skills & experience depth
        if not self._strong_skill_match(resume, job):
            return self._conditional_match("Skill gaps detected")
        
        # Stage 3: Soft signals & potential
        score, insights = self._deep_evaluation(resume, job)
        
        if score >= 90:
            return {"decision": "Strong Hire", "insights": insights}
        # ... more nested/structured logic
```

This becomes the intelligent core of your application, easily extensible with new stages.

## 11. Complexity Analysis

- Time: Still O(1) per path taken, but deeper nesting can increase average evaluation time if many conditions are checked.
- Space: O(1) additional, but cognitive complexity grows with depth (measured by tools like `radon`).

## 14. Code Review Checklist

- **Avoid**: Nesting deeper than 3 levels, duplicated conditions, magic numbers in conditions.
- **Follow**: Early returns/guard clauses, extracted helper methods, clear separation of concerns.
- **Naming**: `meets_hard_filters()`, `_assess_skill_depth()`, etc.
- **Interviewers expect**: Recognition that deep nesting is a code smell, and ability to refactor it into maintainable code.
