**Match Case**

## 1. Introduction

The `match-case` statement (introduced in Python 3.10) provides structural pattern matching, similar to switch-case in other languages but far more powerful. It exists to handle complex branching logic in a clean, readable, and expressive way. In ML engineering, it is highly valuable for parsing configurations, handling different model types, data formats, and inference routing with minimal boilerplate.

## 2. Core Theory

- **Definition**: Structural pattern matching that compares a subject against a series of patterns.
- **Syntax**:
  ```python
  match subject:
      case pattern1:
          # code
      case pattern2 if guard:
          # code
      case _:
          # default
  ```
- **Key Properties**:
  - Supports literal, capture, sequence, mapping, class, and wildcard patterns.
  - Evaluates cases sequentially until a match.
  - Guards (`if` conditions) add extra filters.
  - More powerful and safer than long `if-elif` chains.

## 3. Core Methods / Operations

**Basic Match**
```python
def http_status(code: int) -> str:
    match code:
        case 200:
            return "OK"
        case 404:
            return "Not Found"
        case _:
            return "Unknown"
```

**Advanced Patterns**
```python
match config:
    case {"model": "resnet", "layers": int() as n} if n > 50:
        return "Large ResNet"
    case {"optimizer": "adam", **rest}:
        return f"Adam config: {rest}"
    case list() as batch if len(batch) > 32:
        return "Large batch"
    case _:
        return "Default"
```

**Class & Sequence Patterns**
```python
match model_output:
    case torch.Tensor(shape=(batch, classes)):
        ...
```

## 6. Common Mistakes

1. **Forgetting the wildcard case**  
   Wrong: No `_` case.  
   Correct: Always include `case _:` for safety.

2. **Using match for simple boolean logic**  
   Wrong: Overkill for two conditions.  
   Correct: Use `if-else` or ternary.

3. **Complex patterns without guards**  
   Wrong: Overly broad patterns.  
   Correct: Add `if` guards for precision.

4. **Performance in hot loops**  
   Wrong: Heavy match inside training loop.  
   Correct: Use for configuration/setup only.

5. **Assuming order independence**  
   Wrong: Relying on specific case order incorrectly.  
   Correct: Remember cases are tried top to bottom.

## 7. Practical Examples

**1. Basic Example**
```python
def get_phase(name: str) -> str:
    match name.lower():
        case "train" | "training":
            return "train"
        case "val" | "validation":
            return "val"
        case _:
            raise ValueError(f"Unknown phase: {name}")
```

**2. Intermediate Example**
```python
def parse_hyperparam(param: dict) -> dict:
    match param:
        case {"lr": float() as rate, "optimizer": "adam"}:
            return {"learning_rate": rate, "type": "adam"}
        case {"scheduler": str() as sch}:
            return {"scheduler": sch}
        case _:
            return param
```

**3. Logic Building Example**
```python
def handle_model_output(output, task_type: str):
    match (task_type, output):
        case ("classification", torch.Tensor() as probs) if probs.dim() == 2:
            return probs.argmax(dim=1)
        case ("regression", torch.Tensor() as preds):
            return preds.squeeze()
        case _:
            return output
```

**4. Real Software Example**
```python
def load_checkpoint(checkpoint: dict, framework: str):
    match framework:
        case "pytorch":
            return torch.load(checkpoint["path"], weights_only=True)
        case "tensorflow":
            return tf.saved_model.load(checkpoint["path"])
        case "onnx":
            return onnxruntime.InferenceSession(checkpoint["path"])
        case _:
            raise NotImplementedError(f"Unsupported framework: {framework}")
```

**5. Production / ML Example**
```python
def dispatch_training(config: dict):
    match config:
        case {"model_type": "transformer", "size": "large", **kwargs}:
            model = build_large_transformer(**kwargs)
            trainer = TransformerTrainer(model)
        case {"model_type": "cnn", "backbone": str() as bb}:
            model = build_cnn(backbone=bb)
            trainer = CNNTrainer(model)
        case {"use_distributed": True}:
            trainer = DistributedTrainer(config)
        case _:
            trainer = BaseTrainer(config)
    
    trainer.train()
    return trainer
```

## 8. Time Complexity

Match-case evaluation is generally O(1) for simple patterns and scales with pattern complexity (still efficient for typical ML configs).

| Operation              | Time Complexity |
|------------------------|-----------------|
| Simple literal match   | O(1)            |
| Complex structural match | O(k) (pattern size) |
| Multiple cases         | O(number of cases) worst case |

## 9. Key Takeaways

- Use `match-case` for clean, readable multi-branch logic.
- Leverage powerful patterns (mapping, class, sequence) for data-driven decisions.
- Always include a wildcard (`_`) case.
- Prefer `match` over long `if-elif` chains when handling structured data.
- Excellent for configuration parsing and model dispatching in ML systems.
- Combine with guards for precise control.
- Keep match statements focused — extract complex logic into functions.
- Ideal for production ML code dealing with heterogeneous experiment setups.
- Improves code clarity and reduces bugs compared to deeply nested ifs.
