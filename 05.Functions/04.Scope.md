# **Scope**

## 1. Introduction

**What is this topic?**  
Scope defines the region of a program where a variable (or name) can be accessed. Python uses lexical (static) scoping with the LEGB rule.

**Why does Python have it?**  
Scope enables encapsulation, prevents name collisions, and supports modular code. It is essential for writing safe, maintainable functions and modules.

**What problem does it solve?**  
Without proper scoping, all variables would be global, leading to bugs, unpredictable behavior, tight coupling, and difficulty reasoning about code — especially problematic in large applications and concurrent ML systems.

---

## 2. Intuition

Think of scope like **rooms in a house**:
- Local scope = variables inside a specific room (function).
- Enclosing scope = variables in the outer room (nested function).
- Global scope = variables in the whole house.
- Built-in scope = things available everywhere (like `print`, `len`).

You can see items in your current room easily. To access something from another room, you need explicit permission (e.g., `global`, `nonlocal`).

---

## 3. Syntax

```python
# Global scope
PROJECT_NAME = "Resume Intelligence"

def process_resume(text: str) -> dict:
    # Local scope
    local_var = "temporary"
    
    def inner_helper(section: str) -> str:
        # Enclosing (nonlocal) scope
        nonlocal local_var
        # Can read PROJECT_NAME from global
        return f"{PROJECT_NAME} - {section}"
    
    return inner_helper("skills")

# Built-in scope: len, str, etc.
```

**Key keywords:**
- `global var` — declare intent to modify global.
- `nonlocal var` — modify variable from enclosing scope (Python 3+).
- No declaration needed for reading.

---

## 4. Internal Working

Python follows the **LEGB rule** (Local → Enclosing → Global → Built-in) at runtime when resolving names.

- Each function call creates a new local namespace (stored in `locals()`).
- Closures capture enclosing variables.
- Global namespace lives for the duration of the module.
- Scope is determined at **compile time** (lexical), but lookup happens at runtime.

Modifying globals or enclosing variables requires explicit `global`/`nonlocal` to avoid creating new local variables accidentally.

---

## 5. Built-in Methods / Operations

- `globals()` — returns global namespace dict.
- `locals()` — returns local namespace dict (read-only in most cases).
- `inspect` module for deeper scope introspection.
- `nonlocal` and `global` statements.

---

## 6. Common Mistakes

1. **Unintended local variable creation**:
   ```python
   count = 0
   def increment():
       count += 1  # UnboundLocalError!
   ```

2. Overusing `global` — creates hidden dependencies.

3. Modifying mutable objects from outer scope without realizing (lists, dicts).

4. Closures capturing loop variables (classic gotcha).

5. Assuming nested functions can freely modify outer variables.

---

## 7. Best Practices

- Minimize use of `global` and `nonlocal`.
- Prefer returning values over modifying outer state.
- Use classes or modules for shared state when needed.
- Keep functions pure (no side effects on scope).
- Use type hints and clear naming to make scope intent obvious.
- For configuration, consider dependency injection or config objects rather than globals.

---

## 8. Production Usage

- **Modules**: Each Python file has its own global scope.
- **Web frameworks**: Request handlers use local scope heavily; shared state managed via dependency injection.
- **Logging & config**: Often use module-level globals carefully wrapped.
- **Threaded/async code**: Understanding scope prevents race conditions with shared state.
- **Large codebases**: Strict scoping + encapsulation reduces bugs dramatically.

---

## 9. Machine Learning Usage

- **Model training loops**: Local variables for batch data, metrics; avoid globals for model weights.
- **PyTorch / TensorFlow**: Scoping in custom layers, training functions, and data loaders.
- **Scikit-learn pipelines**: Functions maintain local state; global random seeds used sparingly.
- **LLM agents**: Each tool/agent function has isolated scope; shared memory passed explicitly.
- **Data pipelines**: Closures and nested functions for transforms while keeping scope clean.

## 11. Complexity Analysis

- Name lookup via LEGB: Effectively O(1) in practice (hash table).
- Deep nesting increases cognitive complexity (avoid > 2-3 levels).
- Excessive globals increase debugging and testing complexity.
