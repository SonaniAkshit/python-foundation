# **Return**

## 1. Introduction

**What is this topic?**  
The `return` statement sends a value (or multiple values) back from a function to the caller and immediately exits the function.

**Why does Python have it?**  
It enables functions to compute results and pass them for further processing, which is fundamental to building composable, testable, and maintainable code.

**What problem does it solve?**  
Without `return`, functions can only produce side effects (e.g., printing, modifying globals). `return` allows pure functions, proper data flow, chaining operations, and clear separation between computation and I/O — critical for production systems and ML pipelines.

---

## 2. Intuition

Think of a function as a **vending machine**.  
You insert inputs (parameters), the machine does work, and then it **returns** your snack (the result).  
If there’s no snack ready or something goes wrong, it might return nothing (`None`) or an error. Once the snack is dispensed (`return`), the machine stops processing that transaction.

---

## 3. Syntax

```python
def calculate_metrics(resume_text: str) -> dict[str, float]:
    """Compute various resume quality scores."""
    if not resume_text:
        return {"error": "Empty text"}  # Early return

    word_count = len(resume_text.split())
    # ... more computation

    return {
        "word_count": word_count,
        "readability_score": 85.5,
        "skills_density": 0.23
    }
```

**Key syntax:**
- `return expression` — returns the value.
- `return` (alone) or no return → returns `None`.
- Multiple returns (early exits) are common and encouraged for clarity.
- Return multiple values: `return a, b, c` (actually returns a tuple).
- Type hints with `->` show expected return type.

---

## 4. Internal Working

- When `return` is hit, Python evaluates the expression, packs it (if multiple values → tuple), destroys the current stack frame, and passes the object reference back to the caller.
- No further code after `return` executes in that call.
- Returning large objects is cheap (reference, not copy) due to Python’s memory model.
- Functions without explicit return implicitly return `None`.

---

## 5. Built-in Methods / Operations

- `return` is a statement, not a method.
- Related:
  - `yield` (for generators — different topic).
  - `return` inside `try/finally` — `finally` still executes.
  - Pattern: Return `None`, sentinel values, or rich objects (dataclasses, NamedTuple, Pydantic models).

---

## 6. Common Mistakes

1. **Returning mutable objects without copying** when caller might modify them unintentionally.

2. **Multiple responsibilities** in one function with complex return logic.

3. **Ignoring return values** (calling functions with important returns as statements).

4. **Deeply nested returns** making control flow hard to follow.

5. **Returning raw tuples** instead of named structures (hard to maintain).

6. **Using return for flow control** instead of exceptions for true errors.

---

## 7. Best Practices

- Return early for edge cases (guard clauses).
- Return rich types (dataclasses, TypedDict, Pydantic models) instead of plain dicts/tuples when possible.
- Be consistent with return types (use `Optional[]`, unions, or custom Result classes).
- Keep functions pure when feasible — same input, always same output.
- Document return shape in docstring.
- Use type hints religiously.

---

## 8. Production Usage

- **APIs**: Return JSON-serializable dicts or Pydantic models.
- **ETL pipelines**: Each stage returns processed data or a status object.
- **Error handling**: Return `Result[T, Error]` pattern (or raise exceptions).
- **Async functions**: `async def` returning `Awaitable`.
- **CLI tools**: Functions return exit codes or structured data for further formatting.

---

## 9. Machine Learning Usage

- **Model inference**: `predict()` returns probabilities, embeddings, or class labels.
- **Scikit-learn**: `fit()` returns `self` (for chaining), `predict()` returns arrays.
- **PyTorch**: `forward()` returns tensors; training loops collect losses/metrics.
- **LLM pipelines**: Functions return parsed JSON, token counts, or structured `ResumeAnalysis` objects.
- **Evaluation**: Metric functions return scores, often as dicts with multiple metrics.

## 11. Complexity Analysis

- `return` itself: O(1) (just reference passing).
- Returning large data structures: Cost is in creating the object, not the return.
- Multiple early returns can improve average-case time complexity by avoiding unnecessary work.
