# **Keyword Arguments**

## 1. Introduction

**What is this topic?**  
Keyword arguments (kwargs) allow you to pass arguments to functions by explicitly naming the parameter, rather than relying on position.

**Why does Python have it?**  
Python prioritizes readability and clarity. Keyword arguments make function calls self-documenting, reduce errors from wrong argument order, and work seamlessly with defaults.

**What problem does it solve?**  
Long parameter lists become error-prone with only positional arguments. Keyword arguments improve API usability, make refactoring safer, and are essential for functions with many optional parameters — common in production and ML code.

---

## 2. Intuition

Imagine configuring a car:  
- Positional: `car("red", "sedan", True, 2025)` — easy to mix up.  
- Keyword: `car(color="red", body_type="sedan", electric=True, year=2025)` — instantly clear what each value means.

Keywords turn mysterious calls into readable, explicit ones.

---

## 3. Syntax

```python
def analyze_resume(
    text: str,
    *,
    # Keyword-only parameters
    extract_skills: bool = True,
    extract_experience: bool = True,
    confidence_threshold: float = 0.7,
    llm_model: str = "gpt-4o-mini",
    include_warnings: bool = False
) -> ParseResult:
    """Analyze resume with clear keyword configuration."""
    ...
```

**Key syntax:**
- Call: `analyze_resume(resume_text, extract_skills=False, llm_model="claude")`
- Keyword-only: `*` separator (all following parameters must be passed by keyword).
- `**kwargs`: Capture arbitrary keyword arguments.
- Mixing: positional first, then keywords.

---

## 4. Internal Working

When calling a function:
- Python matches positional arguments first.
- Remaining arguments are matched by name (keywords).
- `**kwargs` collects any unmatched keyword arguments into a dict.
- Keyword arguments are passed as name-value pairs; order in the call doesn't matter (after positionals).

This lookup uses the function’s parameter table. Keyword arguments provide clarity at the cost of slight runtime overhead (negligible).

---

## 5. Built-in Methods / Operations

- `**` unpacking: `func(**config_dict)`
- `inspect.signature()` to see parameters.
- `functools.partial()` to pre-bind keywords.
- `**kwargs` in function definitions for flexibility.

---

## 6. Common Mistakes

1. **Using keywords for everything** unnecessarily (when positional is clearer).

2. **Forgetting `*`** for keyword-only, allowing accidental positional misuse.

3. **Overusing `**kwargs`** — hides interface and breaks type checking.

4. **Order issues**: Cannot use keyword before all positionals are satisfied.

5. **Shadowing** built-in names or using poor keyword names.

---

## 7. Best Practices

- Use keywords for boolean flags and optional config.
- Make important parameters keyword-only with `*`.
- Combine with defaults for powerful, readable APIs.
- Prefer explicit parameters over `**kwargs` when possible.
- Use type hints and docstrings to document expected keywords.
- Unpack dicts with `**` for clean configuration passing.

---

## 8. Production Usage

- **FastAPI / Flask**: Query params, body fields as keyword-style.
- **Configuration**: Pass `**model_config` or `**training_kwargs`.
- **Libraries**: Almost every mature Python library uses keywords heavily (e.g., pandas, matplotlib).
- **Testing**: `pytest` fixtures and parametrized tests use keywords.
- **Orchestration**: Workflow functions accept many keyword options.

---

## 9. Machine Learning Usage

- **PyTorch**: `optimizer.step(closure=...)`, DataLoader with many keyword args.
- **Scikit-learn**: `model.fit(X, y, sample_weight=..., **fit_params)`
- **Hugging Face Transformers**: Dozens of keyword args for `pipeline()`, `TrainingArguments`.
- **LangChain / LLM tools**: Prompt templates and chain invocations use keywords extensively.
- **Resume Intelligence**: Perfect for configuring different parsing modes, LLM providers, thresholds.

## 11. Complexity Analysis

- Keyword argument matching: O(k) where k is number of keywords (hash lookup).
- `**kwargs` creation: O(m) for m extra keywords.
- Improves code maintainability more than it affects runtime performance.
