# ***args**

## 1. Introduction

**What is this topic?**  
`*args` allows a function to accept a variable number of positional arguments, which are collected into a tuple.

**Why does Python have it?**  
It provides flexibility for functions that need to handle an unknown number of inputs without forcing callers to use lists or tuples explicitly. This is a core part of Python’s “batteries included” philosophy for building expressive APIs.

**What problem does it solve?**  
It solves the rigidity of fixed parameters when the number of inputs varies (e.g., summing numbers, processing multiple resumes, logging messages). It enables cleaner, more Pythonic code for variadic functions.

---

## 2. Intuition

Think of `*args` like a **shopping bag**:  
You don’t know how many items the customer will bring. The function takes whatever items (arguments) are handed over, packs them into a single bag (tuple), and processes them together. The star `*` “unpacks” the items from the call site into the bag.

---

## 3. Syntax

```python
def process_resumes(*resumes: str | Path | dict, **options) -> list[ParseResult]:
    """Process any number of resumes."""
    results = []
    for resume in resumes:  # resumes is a tuple
        results.append(parse_single(resume, **options))
    return results

# Calling
process_resumes(resume1, resume2, resume3, mode="hybrid")
process_resumes(*list_of_resumes)  # unpacking
```

**Key syntax:**
- `*args` (conventional name; can be `*items`)
- Can combine with regular parameters: `def func(a, b, *args)`
- Keyword-only after: `def func(*args, keyword_only=True)`
- Type hints: `*args: int` (applies to each element)

---

## 4. Internal Working

- When the function is called, Python collects all extra positional arguments into a new tuple bound to the `*args` name.
- This tuple is created in the local scope.
- Unpacking with `*` at call site iterates the iterable and passes each element as a separate positional argument.
- `*args` appears after fixed positional parameters and before keyword-only ones in the signature.

---

## 5. Built-in Methods / Operations

- `args` is a regular tuple → supports indexing, iteration, slicing, `len()`, etc.
- `*` unpacking in calls and assignments: `first, *middle, last = seq`
- `functools.partial` and other higher-order functions often work well with `*args`.

---

## 6. Common Mistakes

1. **Using `*args` when fixed parameters would be clearer** — hurts readability and type checking.

2. **Forgetting that `args` is a tuple** (trying to append to it).

3. **Placing `*args` in the wrong position** in the signature (after defaults or keyword-only).

4. **Excessive use of `*args` and `**kwargs`** together without clear purpose (makes APIs opaque).

5. **Performance issues** with very large `*args` (creates tuple copy).

---

## 7. Best Practices

- Use `*args` sparingly and only when the number of arguments is truly variable.
- Prefer explicit parameters or accepting a collection (list) when possible.
- Name it descriptively: `*resumes`, `*values`, `*files`.
- Combine with keyword-only parameters for clarity.
- Document what types are expected inside `*args`.
- Consider `typing.ParamSpec` or protocols for advanced type hinting in modern Python.

---

## 8. Production Usage

- **Logging**: `logger.log(level, *messages)`
- **Data pipelines**: Processing variable number of data sources.
- **Command-line tools**: Aggregating multiple input files.
- **Mathematical utilities**: `sum_all(*numbers)`
- **Testing**: Parameterized tests that accept variable inputs.
- **API wrappers**: Forwarding arguments to underlying libraries.

---

## 9. Machine Learning Usage

- **Data loaders**: Handling variable batch sizes or multiple datasets.
- **Ensemble methods**: `ensemble_predict(*model_predictions)`
- **Loss functions**: Combining multiple loss terms `*losses`.
- **PyTorch**: Custom collate functions often use `*args`.
- **LLM tools**: Processing multiple documents or chunks: `analyze_documents(*resume_texts)`.

## 11. Complexity Analysis

- Creating the `*args` tuple: O(n) where n is the number of extra arguments.
- Iteration over `*args`: Standard tuple iteration.
- Unpacking large iterables has memory cost — prefer explicit collections for very large inputs.
