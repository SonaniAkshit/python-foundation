# **Function Basics**

## 1. Introduction

**What is this topic?**  
Functions are the fundamental building blocks of reusable, organized, and maintainable code in Python. A function is a named block of code that performs a specific task and can be executed (called) multiple times from different parts of your program.

**Why does Python have it?**  
Python follows the DRY (Don't Repeat Yourself) principle. Functions allow you to encapsulate logic, reduce duplication, improve readability, and make testing easier. They are central to modular programming and are the foundation for larger constructs like classes, modules, and packages.

**What problem does it solve?**  
Without functions, even medium-sized programs become a long, unreadable sequence of statements (spaghetti code). Functions solve code duplication, improve debuggability, enable code reuse across projects, and support separation of concerns — critical for production applications and ML pipelines.

---

## 2. Intuition

Think of a function like a **kitchen appliance** (e.g., a blender).  
- You put ingredients in (parameters/arguments).  
- It performs a specific job (the code inside).  
- It gives you a result (return value) or just does the job (side effects).  
You don’t need to know *how* the blender works internally every time — you just use it. This abstraction lets you focus on the recipe (your main program) instead of reinventing blending logic every time.

---

## 3. Syntax

```python
def function_name(parameter1: int, parameter2: str = "default") -> return_type:
    """Docstring explaining what the function does."""
    # Function body
    result = parameter1 + len(parameter2)
    return result  # Optional; functions return None by default
```

**Key elements:**
- `def` keyword
- Function name (snake_case per PEP 8)
- Parameters (can have defaults, type hints)
- `->` return type annotation (Python 3.5+)
- Docstring (triple quotes) — highly recommended
- `return` statement

**Calling a function:**
```python
result = function_name(42, "hello")
```

**Important variations:**
- Positional arguments
- Keyword arguments: `func(param1=10)`
- `*args` for variable positional arguments
- `**kwargs` for variable keyword arguments
- Lambda functions: `lambda x: x * 2`

---

## 4. Internal Working

When you call a function:
1. Python creates a new **stack frame** (local namespace).
2. Arguments are passed (by object reference — "pass-by-assignment").
3. The function body executes in this isolated scope.
4. On `return`, the stack frame is destroyed and the value is passed back.
5. Default arguments are evaluated **once** at definition time (mutable defaults like `[]` are a common gotcha).

This stack-based execution enables recursion and proper variable scoping.

---

## 5. Built-in Methods / Operations

Functions themselves are **first-class objects** in Python.

Important operations:
- `help(function)` or `function.__doc__` — access docstring
- `function.__name__` — get name
- `callable(obj)` — check if something is callable
- `functools.partial()` — create specialized versions
- `map()`, `filter()`, `reduce()` — functional style (though list comprehensions are often preferred)

---

## 6. Common Mistakes

1. **Mutable default arguments**:
   ```python
   def bad_func(items=[]):  # Wrong
       items.append(1)
       return items
   ```
   Fix: `def good_func(items=None): if items is None: items = []`

2. **Forgetting `return`** — function silently returns `None`.

3. **Modifying global variables** without `global` (avoid globals anyway).

4. **Overusing `*args`/`**kwargs`** — makes code harder to understand and type-check.

5. **Deep nesting** or giant functions — violates single responsibility principle.

---

## 7. Best Practices

- Keep functions small (< 50 lines ideally).
- One clear responsibility per function.
- Use descriptive names (`extract_resume_skills()` not `process()`).
- Always write docstrings.
- Use type hints (helps IDEs, mypy, and future readers).
- Prefer explicit parameters over `*args`/`**kwargs` when possible.
- Follow the "Command-Query Separation" principle where appropriate.

---

## 8. Production Usage

- **Backend APIs**: Each endpoint handler is typically a function (FastAPI, Flask).
- **Automation scripts**: ETL jobs broken into `extract()`, `transform()`, `load()`.
- **CLI tools**: Click/Typhoon commands wrap functions.
- **Testing**: Every unit test is built around calling functions predictably.
- **Error handling**: Functions often raise custom exceptions or return `Result` types.

---

## 9. Machine Learning Usage

- **Data pipelines**: `preprocess_text(resume)`, `extract_features()`.
- **Scikit-learn**: Custom transformers inherit and implement `fit()`/`transform()`.
- **PyTorch**: Model forward pass is a method, but helper functions handle data loading/augmentation.
- **LLM applications**: Prompt engineering functions, parsing responses, evaluation metrics.
- **Pandas**: Many operations (`df.apply()`) accept functions.

## 11. Complexity Analysis

- Function definition: O(1) (just creates a function object).
- Function call overhead: Small but non-zero (stack frame creation).
- Recursion depth limited by `sys.getrecursionlimit()` (default 1000) — watch for stack overflow.

## 14. Code Review Checklist

- Function does one thing (Single Responsibility).
- Parameters are clearly named and typed.
- Docstring is present and useful.
- No mutable defaults.
- Error handling is appropriate (or let exceptions bubble up).
- Pure functions preferred over heavy side effects.
- Naming follows PEP 8 (verbs for functions: `get_`, `parse_`, `compute_`).
- Interviewers look for: clarity, edge cases, type safety, and testability.
