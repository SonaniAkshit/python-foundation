# **Parameters**

## 1. Introduction

**What is this topic?**  
Parameters are the mechanism by which we pass data into functions. They define the interface of a function — what inputs it expects and how it can be used.

**Why does Python have it?**  
Python is designed for readability and flexibility. Its parameter system supports positional, keyword, default, variable, and annotated arguments, making functions both powerful and ergonomic.

**What problem does it solve?**  
It allows functions to be configurable and reusable without hard-coding values. Proper parameter design prevents tight coupling, improves API usability, and makes code testable and extensible — essential for large codebases and ML systems.

---

## 2. Intuition

Imagine ordering coffee at a café.  
- The barista’s “function” has parameters: `size`, `drink_type`, `milk_option`, `extras`.  
- You can use positional order (“large latte oat”), keyword (“size='large'”), or defaults (“regular milk” if not specified).  
- `*extras` lets you add any number of toppings.  

Parameters turn a rigid procedure into a flexible, user-friendly service.

---

## 3. Syntax

```python
def process_resume(
    file_path: str,                    # required positional
    extract_skills: bool = True,       # default
    custom_keywords: list[str] | None = None,  # modern union syntax
    *args: Any,                        # variable positional
    **kwargs: Any                      # variable keyword
) -> dict:
    """Process resume with flexible parameters."""
    ...
```

**Key syntax (Python 3.13+):**
- Positional-only: `def func(a, b, /)`
- Keyword-only: `def func(*, c, d)`
- Default values
- Type hints with `|` for unions (3.10+)
- `*args`, `**kwargs`
- `/` and `*` separators for control

---

## 4. Internal Working

- Parameters become local variables in the function’s stack frame.
- Arguments are passed by **assignment** (references to objects).
- Defaults are created **once** at `def` time → mutable default trap.
- Python uses the following order when matching arguments:
  1. Positional
  2. Keyword
  3. Variable `*args`
  4. Keyword-only
  5. `**kwargs`

Type hints are **not enforced** at runtime (use `typing` + `mypy` or `pydantic` for validation).

---

## 5. Built-in Methods / Operations

- `inspect.signature(func)` — introspect parameters at runtime.
- `functools.partial(func, arg1=val)` — freeze some parameters.
- `dataclasses` / `pydantic` models for complex parameter groups.
- `**` unpacking: `func(**resume_dict)`

---

## 6. Common Mistakes

1. **Mutable defaults**:
   ```python
   def bad(resume_data: dict = {}):  # Dangerous!
       resume_data["processed"] = True
   ```

2. Using too many positional arguments → hard to remember order.

3. Mixing `*args` and `**kwargs` without clear purpose.

4. Not using keyword-only parameters for optional flags.

5. Ignoring type hints → runtime surprises in large projects.

---

## 7. Best Practices

- Use meaningful names (verbs for actions, nouns for data).
- Prefer explicit parameters over `*args`/`**kwargs`.
- Put required parameters first, then defaults.
- Use keyword-only for boolean flags and optional configs.
- Always add type hints.
- Keep total parameters reasonable (< 6–7); consider a config object/dataclass for more.
- Document constraints in docstring.

---

## 8. Production Usage

- **FastAPI / Flask**: Path, Query, Body parameters with Pydantic models.
- **CLI tools** (Typer/Click): Automatic parameter parsing.
- **Data pipelines**: Functions with `config: PipelineConfig` dataclass.
- **Error handling & retries**: Parameters for `max_retries`, `timeout`, `backoff_factor`.
- **Logging & observability**: `trace_id`, `user_id` as keyword-only.

---

## 9. Machine Learning Usage

- **Scikit-learn** estimators: `fit(X, y, sample_weight=None, **fit_params)`
- **PyTorch** `DataLoader`: `batch_size`, `shuffle`, `num_workers`, `collate_fn`
- **Hugging Face**: Model loading functions with dozens of optional parameters (`trust_remote_code`, `device_map`, etc.).
- **LLM pipelines**: `temperature`, `max_tokens`, `top_p`, `system_prompt`.
- **Pandas**: Many methods accept flexible `**kwargs`.

## 11. Complexity Analysis

- Parameter passing: O(1) per argument (reference copy).
- `*args` / `**kwargs` creation: O(n) where n is number of extra args.
- Deep parameter lists increase cognitive complexity — keep functions focused.
