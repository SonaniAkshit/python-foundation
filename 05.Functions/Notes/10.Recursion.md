# **Recursion**

## 1. Introduction

**What is this topic?**  
Recursion is a programming technique where a function calls itself to solve smaller instances of the same problem.

**Why does Python have it?**  
It naturally expresses divide-and-conquer algorithms, tree/graph traversals, and problems with self-similar structure. Python supports recursion as a first-class control flow mechanism.

**What problem does it solve?**  
It simplifies code for problems that have recursive structure (e.g., directory traversal, parsing nested data, backtracking), making solutions more elegant and easier to reason about compared to complex iterative stacks.

---

## 2. Intuition

Imagine Russian nesting dolls (Matryoshka).  
To open the largest doll, you recursively open the smaller one inside it until you reach the tiniest doll (base case). Then you work your way back out. Recursion breaks a big problem into identical smaller problems until a trivial case is reached.

---

## 3. Syntax

```python
def factorial(n: int) -> int:
    """Compute factorial recursively."""
    if n <= 1:          # Base case
        return 1
    return n * factorial(n - 1)  # Recursive case

def safe_recursive_parse(node: dict, depth: int = 0) -> dict:
    if depth > 20 or not node:  # Safety base case
        return {}
    # Process current + recurse on children
    return {
        "data": node.get("content"),
        "children": [safe_recursive_parse(child, depth + 1) for child in node.get("sections", [])]
    }
```

**Key elements**:
- **Base case** (stop condition) — mandatory to prevent infinite recursion.
- **Recursive case** — call with smaller input.
- Tail recursion (last operation is the recursive call) — Python does not optimize it automatically.

---

## 4. Internal Working

Each recursive call creates a new stack frame with its own local variables.  
- The call stack grows until base cases are hit, then unwinds with returns.
- Python has a default recursion limit (`sys.getrecursionlimit()` ≈ 1000) to prevent stack overflow.
- Memory usage is O(depth) for stack frames; time complexity depends on the recurrence relation (e.g., T(n) = T(n-1) + O(1) for linear recursion).

---

## 5. Built-in Methods / Operations

- `sys.setrecursionlimit(n)` — adjust limit (use cautiously).
- `functools.lru_cache()` — memoization decorator for expensive recursive functions.
- `itertools` and explicit stacks for converting recursion to iteration when needed.

---

## 6. Common Mistakes

1. **Missing or incorrect base case** → infinite recursion / RecursionError.

2. **No progress toward base case** (e.g., not reducing input size).

3. **Exceeding recursion depth** on large inputs.

4. **Inefficient recursion** without memoization (exponential time).

5. **Using recursion when iteration is simpler and more efficient** (e.g., factorial, Fibonacci).

---

## 7. Best Practices

- Always define clear, reachable base cases.
- Keep recursive functions pure when possible.
- Use memoization (`@lru_cache`) for overlapping subproblems.
- Consider iterative alternatives or explicit stack for deep recursion.
- Add depth limiting or safety checks in production code.
- Document recursion depth expectations.

---

## 11. Complexity Analysis

- **Time**: Depends on recurrence (e.g., O(n) for linear, O(2^n) for naive Fibonacci).
- **Space**: O(recursion depth) due to call stack (can be optimized with tail recursion in other languages, but not Python).
- Always analyze both time and space before using in production.
