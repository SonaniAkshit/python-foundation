**Lambda**

---

## 1. Introduction

**What is this topic?**  
Lambda functions are small, anonymous (unnamed) functions defined with the `lambda` keyword. They are single-expression functions.

**Why does Python have it?**  
Lambda provides a concise way to create throwaway functions for short operations, especially useful as arguments to higher-order functions like `map()`, `filter()`, `sorted()`, or in functional-style code.

**What problem does it solve?**  
It reduces boilerplate when you need a simple function for a short time, avoiding the overhead of a full `def` statement. In production and ML code, lambdas enable quick transformations, key functions, and callbacks.

---

## 2. Intuition

Think of a lambda as a **temporary vending machine button** — it does one specific job (a single expression) without needing a full machine (named function) built for it. You use it once or pass it to someone else who needs that exact simple behavior.

---

## 3. Syntax

```python
# Basic
square = lambda x: x ** 2

# With multiple parameters
add = lambda x, y: x + y

# Common usage
sorted_resumes = sorted(resumes, key=lambda r: r.get("score", 0), reverse=True)

# With type hints (Python 3.10+)
from typing import Callable
extract_score: Callable[[dict], float] = lambda r: r.get("confidence", 0.0)
```

**Limitations**:
- Single expression only (no statements, no `return` keyword — implicit return).
- No annotations inside the lambda (use variable annotation outside).
- No docstring.

---

## 4. Internal Working

- A lambda creates a function object just like `def`, but without a `__name__` (it’s `<lambda>`).
- The body is an expression evaluated when the lambda is called.
- Lambdas capture variables from enclosing scope (closures) — same rules as nested functions.
- They are syntactic sugar; the bytecode is similar to a regular function.

---

## 5. Built-in Methods / Operations

- Lambdas are first-class: can be assigned, passed, returned, stored in lists/dicts.
- Common with: `map()`, `filter()`, `reduce()` (from `functools`), `sorted(key=)`, `max/min(key=)`.
- `operator` module often preferred for common operations (e.g., `itemgetter` instead of lambda).

---

## 6. Common Mistakes

1. **Overusing lambdas** for complex logic — hurts readability and debuggability.

2. **Trying to put statements** (if, loops, assignments) inside lambda → SyntaxError.

3. **Late binding closures** in loops (classic gotcha: all lambdas capture the final loop value).

4. **Ignoring readability** — long/complex lambdas are worse than named functions.

5. **Not using type hints** on assigned lambdas.

---

## 7. Best Practices

- Use lambdas for very short, obvious operations.
- Prefer named functions for anything non-trivial or reused.
- Keep lambdas on one line.
- Use `operator` module or comprehensions when possible.
- For complex key functions, extract to a named helper.
- In production code, favor readability over cleverness.

---

## 8. Production Usage

- **Data processing / ETL**: Quick transformations in pandas `apply()` or Spark.
- **Sorting / grouping**: Custom keys in large datasets.
- **Event handlers / callbacks**: Simple listeners.
- **Configuration**: Dynamic key functions.
- **Testing**: Quick mocks or predicates.

---

## 9. Machine Learning Usage

- **Pandas**: `df.apply(lambda x: ...)` for row-wise operations.
- **Scikit-learn**: Custom scorers or transformers.
- **PyTorch**: Simple collate functions or hooks.
- **Data augmentation**: Quick transform lambdas.
- **Resume Intelligence**: `key=lambda resume: resume["score"]` for ranking candidates, simple text cleaners, or filtering.

## 11. Complexity Analysis

- Creation: O(1).
- Execution: Same as regular function for the expression.
- Overuse increases cognitive complexity and debugging difficulty.
